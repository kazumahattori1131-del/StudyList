音声・スライド・動画生成で繰り返し発生したバグと対策をまとめる。
新しいミスが見つかったらここに追記すること。

> **動画生成スクリプト（slide_to_video.py）は起動時にこのファイルのチェックリストを自動表示する。**

## 0. 生成前チェックリスト ← 毎回ここを確認してから生成する

新しい動画の voice.md / HTML を書き終えたら、以下を必ず確認する。

- [ ] `GEMINI_TTS_VOICE = 'Leda'` のまま変わっていないか・起動バナーの「TTS 設定確認」が全て OK になっているか（PITFALLS.md 3-8）
- [ ] 類題スライド：問題文と解説が **空行1行** で正しく区切られているか（問題文は空行の前に入れる）
- [ ] 類題の変数名が HTML と一致しているか（HTML が $T$ なのに「エス」と読んでいないか）
- [ ] 数式の操作の言い方が正しいか（「割る」と「かける」を混同していないか）
- [ ] 数学用語の助詞が正しいか（「積**を**分解」ではなく「積**に**分解」など）
- [ ] 助数詞が正しく書かれているか（2つ/三つ NG → ふたつ/みっつ OK）
- [ ] 「+…+」ではなく「プラスてんてんてんプラス」と書いているか
- [ ] HTML 暗背景ボックスに `color: #fff` が付いているか
- [ ] HTML に `@supports` フォールバック色が書かれているか（color-mix() 非対応対策）

## 1. 音声台本（_voice.md）

### 1-1. 類題スライドの分割構造

`split_ruidai_script()` は **最初の空行（`\n\n`）** で台本を2分割する。
- 空行より前 → 問題提示（解答非表示）
- 空行より後 → 解説（解答表示）

**NG：** 問題文を2段落目に書く（解説側に入ってしまう）
```
では類題です。          ← ここだけ problem になる

nを正の整数として...    ← solution 側に入り、解答が見えた状態で読まれる
```

**OK：** 問題文を問題パート内に1段落でまとめる
```
では類題を解いてみましょう。
nを正の整数として...    ← 問題文もここに含める（改行はOK、空行はNG）

今度の公比は3なので...   ← ここから解説（空行で区切る）
```

コードが自動で `\nでは一度、自分で解いてみてください。` を末尾に付加するので、
voice.md には「考えてみてください」系の文を書かなくてよい。

---

### 1-2. 数式の操作の言い方（割る vs かける）

分数を含む等式を整理するとき、「両辺を割る」ではなく「両辺にかける」が正しい場合がある。

例：`-2T = (3^{n+1} - 3)/2 - n·3^{n+1}` から `4T = ...` を導く操作は
**両辺に `-2` をかける** であり、「マイナス2で割る」ではない。

| NG | OK |
|---|---|
| 「両辺をマイナス2で割って整理すると」 | 「両辺にマイナス2をかけて整理すると」 |

台本を書く前に、実際の式変形の操作（掛ける・割る・移項など）を確認する。

---

### 1-3. 助詞の違いに注意（積を分解 vs 積に分解）

| NG | OK |
|---|---|
| 「シグマは積を分解できません」 | 「シグマは積に分解できません」 |

「積を分解する」→ 積（かけ算）を分割するという意味になり、意図と異なる。
「積に分解する」→ Σ を 積の形（Σa × Σb）に分解する、という正しい意味。

数学用語は助詞ひとつで意味が変わるため、HTML のテキストと voice.md を照合して確認する。

---

### 1-4. 変数名の読み仮名

類題で使う変数が HTML と一致しているか確認する。

| 状況 | NG | OK |
|---|---|---|
| HTML が $T$ を使っているのに | 「エスと3エスを並べて」 | 「ティーと3ティーを並べて」 |

---

### 1-5. 助数詞の表記

normalize_for_tts() で変換されるが、voice.md 側も正しく書いておく。

| NG | OK |
|---|---|
| 二つ / 2つ | ふたつ |
| 三つ / 3つ | みっつ |
| 1問 | いちもん（「問」→「もん」は自動変換） |

---

### 1-7. 類題の解答部分（solution_script）内の空行も長い間を生じさせる

`split_ruidai_script()` は最初の `\n\n` で問題／解答を分割するが、
解答部分の**内側に空行が残っていると**、TTS がその箇所で不自然に長い間を置く。

**NG:**
```
t イコール 3のx乗 とおきます。
3のx乗 イコール 5 より、x イコール 3 を底とする 5 の対数。

手順は同じです。自分でも確認してみてください。   ← 空行が間を生む
```

**OK:** 解答部分内の空行もすべて削除する
```
t イコール 3のx乗 とおきます。
3のx乗 イコール 5 より、x イコール 3 を底とする 5 の対数。
手順は同じです。自分でも確認してみてください。
```

voice.md では `…` を使わず、明示的に書く。
normalize_for_tts() で変換されるが、voice.md に直書きする方が確実。

| NG | OK |
|---|---|
| `3の1乗＋…＋3のn乗` | `3の1乗プラスてんてんてんプラス3のn乗` |

## 2. normalize_for_tts()（slide_to_video.py）

### 2-1. 省略記号の正規表現

character class `[プラス]` は「プ」「ラ」「ス」の**1文字ずつ**にマッチする（単語ではない）。

**NG:**
```python
re.sub(r'[…][\s　]*[プラス＋+]', 'てんてんてんプラス', text)
```

**OK:**
```python
re.sub(r'[…](?:\s*(?:プラス|[＋+]))', 'てんてんてんプラス', text)
```

### 2-2. 助数詞変換は数字変換より先に行う

`('2つ', 'ふたつ')` の置換を先に行わないと、先に `2` → `に` に変換されて `につ` になる。

### 2-3. 正規表現の負の先読みで「正し（い/く）」を除外する

`正(?!し)` だと「正しい」も「正しく」も除外されず「せいしい」と読まれる。
`正(?!し|式|規|確|解|直|午|月|論|比|弦|接|反|逆|面|答|法)` のように `し` を1文字だけ
負の先読みに入れれば「正し…」系は全て除外される。

## 3. Gemini TTS API

### 3-1. system_instruction は TTS モデル非対応

`system_instruction` を渡すと、モデルがテキスト生成しようとして **500 INTERNAL** または
**400 INVALID_ARGUMENT（"Model tried to generate text"）** が返る。

**OK:** `system_instruction` を削除し、`response_modalities=['AUDIO']` のみ指定する。

### 3-8. GEMINI_TTS_STYLE_PREFIX は定義するだけでは不十分（定義≠適用）

`GEMINI_TTS_STYLE_PREFIX` などのスタイル定数を定義しても、
`generate_audio_gemini` の `contents` に実際に付加しないと TTS に反映されない。
別セッションで「定義されているから適用済み」と誤認されやすい典型的な落とし穴。

**NG（定義するだけで使っていない）：**
```python
GEMINI_TTS_STYLE_PREFIX = "..."   # ← 定義はあるが使われていない
resp = client.models.generate_content(
    contents=normalize_for_tts(text),  # ← スタイル前置きなし
    ...
)
```

**OK（必ず contents に前置きする）：**
```python
contents = GEMINI_TTS_STYLE_PREFIX + normalize_for_tts(text)
resp = client.models.generate_content(contents=contents, ...)
```

**再発防止策：** 新セッション開始時に `generate_audio_gemini` の `contents=` 行を
読み、`GEMINI_TTS_STYLE_PREFIX` が実際に含まれているか必ず目視確認する。

`'500' in err or 'INTERNAL' in err` もリトライ条件に追加する。

### 3-3. 短すぎるテキストで 400 エラーが出ることがある

テスト時は2文以上（「テスト。これはテストです。」）を使う。

### 3-6. 命令文（〜求めよ）を日本語括弧で囲むと 400 エラーが出ることがある

「〜の値を求めよ。」のように、数学問題の問題文が **「」で囲まれた命令形** になっていると
TTS モデルがテキスト生成指示と誤認して `400 INVALID_ARGUMENT: Model tried to generate text` が返る。

**NG:**
```
「2x²-5x+3=0 の2解を α,β とするとき、1/α + 1/β の値を求めよ。」
```

**OK（括弧なし、または宣言形に言い換え）：**
```
2x²-5x+3=0 の2解を α,β とするとき、1/α + 1/β の値を求める問題です。
```

### 3-7. TTS レスポンスの audio data が None になることがある

モデルが text part を先頭に置いて audio part を後続させる場合、
`resp.candidates[0].content.parts[0].inline_data.data` が `None` になる。

**OK:** 全 parts を走査して最初に `inline_data.data` が存在するものを使う。
```python
raw = None
for part in resp.candidates[0].content.parts:
    idata = getattr(part, 'inline_data', None)
    if idata and getattr(idata, 'data', None):
        raw = idata.data
        break
if raw is None:
    raise ValueError('TTS response contained no audio data')
```

### 3-4. APIキーをコード・スクリプトに直接書かない

シェルスクリプトや Python コードに APIキーをハードコードすると、
git にコミットした瞬間に Google に「漏洩キー」と判定されて即時無効化される。

**NG:**
```bash
KEY="AIzaSy..."   # ← コードに直書き
```

**OK:** 環境変数から読む
```bash
KEY="${GEMINI_API_KEY:?'GEMINI_API_KEY が未設定です'}"
```

実行時に `GEMINI_API_KEY=xxx python3 tools/slide_to_video.py ...` のように渡す。

### 3-5. クォータは 100リクエスト/日（課金していても同様）

`gemini-2.5-flash-preview-tts` はプレビューモデルのため、**課金・残高があっても** 上限は 100回/日。
Google Cloud Console の Quotas ページで手動申請すれば増やせる（承認に数日かかる場合あり）。
リセットは毎日 **UTC 0:00**（日本時間 9:00）。

**音声キャッシュ機能**（generate_audio 内）を活用し、テキスト変更がない限り再生成しない。

### 3-9. OUTRO_TEMPLATE の空行で TTS が途中停止する

`OUTRO_TEMPLATE` に `\n\n`（空行）が含まれていると、TTS が途中で停止し、
空行より後のテキスト（「疑問点や〜」など）が読まれなくなる。

**NG:**
```python
OUTRO_TEMPLATE = """今回は「{title}」について解説しました。

この解法を知っておくだけで…

疑問点や解説してほしい問題があれば、コメントで教えてください。それでは！"""
```

**OK:** 空行をすべて除去して1段落にまとめる
```python
OUTRO_TEMPLATE = """今回は「{title}」について解説しました。この解法を…疑問点や解説してほしい問題があれば、コメントで教えてください。それでは！"""
```

### 3-10. `p の2x乗` / `p のx乗` 表記で TTS が異常長音声を生成する

一般化スライドなどで `p の2x乗` や `p のx乗` と書くと、
Gemini TTS が **10分超・30MB超** の異常に長い音声を生成することがある（原因不明）。
これにより、そのスライド以降の音声が無音・ほぼ無音になる。

**NG:**
```
p の2x乗 プラス a かける p のx乗 プラス b イコール 0
```

**OK:** `エックス` を明示的に書く
```
p の 2エックス 乗 プラス a かける p の エックス 乗 プラス b イコール 0
```

発生した場合は `audio_0X.wav` と `audio_0X.md5` を削除して強制再生成する。

## 5. HTML スライド

### 5-1. color-mix() は Playwright Chromium で未サポートの場合がある

`color-mix(in srgb, ...)` が動かず背景が暗くなる。
`@supports` を使ってフォールバック色を必ず書く。

```css
.problem-box { background: #fff6ef; }   /* フォールバック */
@supports (color: color-mix(in srgb, red 8%, white)) {
  .problem-box { background: color-mix(in srgb, var(--accent) 8%, white); }
}
```

### 5-2. 暗背景ボックスのテキスト色を必ず指定する

`.key-box` など `background: #1c1008` のような暗い背景には `color: #fff` を必ず付ける。
未指定だと継承色（黒）になり、文字が見えなくなる。

### 5-3. KaTeX の \phantom で数式の縦揃えを行う

差し引き法の $2S$ の行を $S$ と揃えるには `\phantom{}` を使う。

```html
$2S = \phantom{1{\cdot}2^1 + {}} 1{\cdot}2^2 + \cdots$
```

## 7. 追い装飾指示（_edit.md アニメーション）

### 7-1. 強調対象はHTMLの実際のテキストから引用する

`slide_to_video.py` は `強調対象` フィールドから LaTeX（`$...$`）を除いた
日本語部分を DOM 検索キーワードとして使い、要素の座標を取得する。
**HTMLに存在しない言い回しを書くと bbox が検出できず、アノテーションがスキップされる。**

**NG（概念を書いてしまっている）：**
```
- 強調対象：`放物線が下に開くので、どこかで必ず $x$ 軸より下になる。`
```
→ HTML のテキストが「下に開く放物線」という語順なので一致しない。

**OK（HTMLのテキストをそのまま引用）：**
```
- 強調対象：`下に開く放物線 。$D < 0$（x 軸と交わらない）`
```
→ HTML に実在する文字列のかけらを含めること。

#### 確認手順
1. HTML ファイルをブラウザや VSCode で開く
2. 強調したい要素の**表示テキスト**をコピーして `強調対象` に貼る
3. LaTeX 部分は `$...$` 形式で残してよい（スクリプトが自動除去する）
4. 日本語テキストが 2〜3文字以上含まれていれば検出できる

### 7-2. LaTeX のみの強調対象は検出不可

`$$a < -1 \quad \text{または} \quad a > 2$$` のように日本語ゼロの場合、
DOM 検索キーワードが抽出できず必ず未検出になる。

**対処：** `強調対象` の末尾や先頭にスライド上の日本語ラベル・説明を付け加える。
```
- 強調対象：`（答え）$a < -1$ または $a > 2$`
```

### 7-3. 対応している方法（方法：フィールド）

| 書き方 | 描画内容 |
|---|---|
| `赤枠` / `四角` | 赤い四角形の枠線 |
| `丸枠` / `丸` / `円` | 赤い楕円の枠線 |
| `矢印` | 要素左上への斜め矢印 |
| `アンダーライン` | 要素下端に左→右へ伸びる赤線（0.5秒アニメーション） |
| `黄色ハイライト` 等 | 未対応のためスキップ（赤枠として描画される） |

### 6-1. QuickTime Player は yuv420p が必要

ffmpeg のデフォルト（yuv444p 等）は QuickTime で再生できない。
`write_videofile()` には必ず以下を渡す。

```python
ffmpeg_params=['-pix_fmt', 'yuv420p', '-movflags', '+faststart']
```

## 8. Claude Code のストリームタイムアウト（"Stream idle timeout"）

### 8-1. エラーの正体

```
API Error: Stream idle timeout - partial response received
```

Claude が一度のレスポンスで **巨大なファイル（数百行の HTML など）を生成しようとする**と、
トークン生成に時間がかかりすぎてストリームがタイムアウトする。

### 8-2. やってはいけないこと

- 1回の応答で 200行超のファイルを `Write` ツールで書こうとする
- `Agent` ツールに「HTMLを1ファイルまるごと書いて」と委任する
  （サブエージェントも同じタイムアウトに引っかかる）

### 8-3. 正しい対処：`cat >> file` の分割チャンク書き込み

大きなファイルは Bash で小さなチャンクに分けて追記する。

```bash
# 1チャンク目（ヘッダ）
cat > output.html << 'CHUNK1'
  ... 30行程度 ...
CHUNK1

# 2チャンク目（スライド①）
cat >> output.html << 'CHUNK2'
  ... 30〜40行程度 ...
CHUNK2

# 以降、スライド1枚ずつ追記
cat >> output.html << 'CHUNK3'
  ...
CHUNK3
```

**目安：1チャンクあたり 40〜50行以内**。各チャンクの後に
`echo "OK: $(wc -l < output.html) lines"` で進捗確認すると安全。

### 8-4. voice.md・edit.md は1回で書いてよい

130行程度のテキストファイルなら `cat > file << 'EOF' ... EOF` で一括書き込みできる。
タイムアウトが起きるのは主に HTML（SVG・KaTeX 入り）の生成時。
